# How `specify init` Works

`specify` is a Python CLI, distributed from the spec-kit repo and run without a separate
install step via `uvx`:

```bash
uvx --from git+https://github.com/github/spec-kit.git specify init --here --integration claude
```

`specify` itself has several subcommands (`specify --help`):

| Subcommand | Purpose |
|---|---|
| `init` | Scaffold a new or existing project — the focus of this doc |
| `check` | Verify required tools are installed |
| `version` | Print CLI/system version info |
| `self` | Check for newer spec-kit releases, preview upgrades (`--dry-run`), upgrade in place |
| `extension` | Manage spec-kit extensions (the hook system — see `markdown-templates.md`) |
| `integration` | Manage coding-agent integrations |
| `preset` | Manage spec-kit presets (pre-bundled template/constitution customizations) |
| `bundle` | Discover, install, and author spec-kit bundles |
| `workflow` | Manage and run automation workflows |

This repo only used `init`, but `self`, `check`, and `preset` are worth knowing about if
you maintain a spec-kit project over time — `specify self upgrade` is how you'd pull in
template/command improvements after spec-kit itself updates.

## `init`'s key flags

```text
specify init my-project                          # new directory
specify init .                                   # current directory, interactive agent choice
specify init --here --integration claude         # current directory, explicit agent
specify init --here --force                      # skip the "directory not empty" confirmation
specify init my-project --preset healthcare-compliance --integration claude
specify init my-project --integration generic --integration-options="--commands-dir .myagent/commands/"
```

- `--integration <agent>` — which coding agent to generate command files for (claude,
  copilot, gemini, codex, generic, and 25+ others). This is the single flag that
  determines the on-disk shape described in `commands.md` and `markdown-templates.md`.
- `--script <sh|ps|py>` — which shell dialect the helper scripts under
  `.specify/scripts/` are written in. This repo used `sh` (see `.specify/init-options.json`).
- `--preset <id>` — installs a pre-built customization bundle (e.g. a constitution and
  extra template sections tuned for a regulated domain) on top of the base scaffold.
- `--force` — skip the confirmation prompt when the target directory isn't empty, merging
  template files into what's already there rather than refusing to run.
- `--ignore-agent-tools` — skip the check that the chosen agent's CLI/tool is actually
  installed (useful in CI or when scaffolding for an agent you don't run locally).

Per `specify init --help`: *"Project files are scaffolded from assets bundled inside the
specify-cli package, so initialization does not need network access and templates match
the installed CLI version."* Nothing is generated by an LLM at this stage — it's a file
copy from the installed package, which is why it's fully reproducible offline.

## What actually happened in this repo

Running the command above produced this exact sequence:

```text
├── Check required tools (ok)
├── Select coding agent integration (claude)
├── Select script type (sh)
├── Install integration (Claude Code)
├── Install shared infrastructure (scripts (sh) + templates)
├── Ensure scripts executable (5 updated)
├── Constitution setup (copied from template)
├── Install bundled workflow (speckit installed)
└── Finalize (project ready)
```

Three steps are worth calling out individually:

**"Install integration (Claude Code)"** is what wrote `.claude/skills/speckit-*/SKILL.md`
— ten files, one per command. See `commands.md` and `markdown-templates.md` for what's
actually inside them and how they got adapted specifically for Claude.

**"Install bundled workflow (speckit installed)"** wrote
`.specify/workflows/workflow-registry.json`, which records a bundled workflow named
`"Full SDD Cycle"`: *"Runs specify → plan → tasks → implement with review gates."* A
*workflow* here is a distinct concept from an individual *command* — it's a named,
pre-defined chain of commands, managed via `specify workflow`, rather than something you
invoke one skill at a time.

**"Constitution setup (copied from template)"** copied
`.specify/templates/constitution-template.md` to `.specify/memory/constitution.md`
verbatim, brackets and all — `/speckit-constitution` is a separate step that fills it in
later (see `PROCESS.md` section 3).

## How `init` tracks what it installed

Two files exist purely for bookkeeping, and both matter if you ever re-run `init` or
`specify self upgrade` later:

**`.specify/init-options.json`** records the choices made this run:

```json
{
  "ai": "claude",
  "ai_skills": true,
  "feature_numbering": "sequential",
  "here": true,
  "integration": "claude",
  "script": "sh",
  "speckit_version": "0.13.5.dev0"
}
```

`feature_numbering: "sequential"` is why our feature directory is `specs/001-task-list-api/`
rather than a timestamp-prefixed name — future features in this repo will be `002-...`,
`003-...`, and so on.

**`.specify/integrations/claude.manifest.json`** is a checksum manifest of every file the
Claude integration installed, with an install timestamp:

```json
{
  "integration": "claude",
  "version": "0.13.5.dev0",
  "installed_at": "2026-07-23T02:58:43Z",
  "files": {
    ".claude/skills/speckit-constitution/SKILL.md": "70a37fbe...",
    ...
  }
}
```

This is how `specify` can later tell the difference between "this skill file is stock"
and "you've hand-edited it" — a prerequisite for safely merging upgrades without
clobbering local customizations.
